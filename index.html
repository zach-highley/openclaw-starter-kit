The 7 Golden Rules steipete

1

Spec before code

Write what you want before building it. Never start a feature without a spec.

2

AGENTS.md in every project

Single biggest force multiplier. Project context that makes agents 10x more effective.

3

300 LOC max per file

Agents make fewer mistakes with smaller files. Hard constraint ‚Äî no exceptions.

4

Tests per feature, same session

Don't switch context. Write tests right after the feature while the agent has full context.

5

CLIs > MCPs > nothing

One line in AGENTS.md beats installing an MCP server. Less context pollution, more composability.

6

Commit early, commit often

Git is your safety net. Small atomic commits on main. No branches for rapid iteration.

7

Plan first, build second

"Ultrathink, give me 3 options" ‚Äî iterate on the plan before typing "y" to build.

Daily Workflow

The typical day of an agentic engineer: Our experience

  1. Open Ghostty ‚Äî 2-4 tabs: one per agent task, one for git/testing
  2. Check agent health ‚Äî openclaw health, openclaw status
  3. Review TODO/memory ‚Äî what's pending, what context carries over
  4. Spec or plan first ‚Äî Gemini for big features, quick prompt for small ones
  5. Build with agents ‚Äî steer every few minutes, keep files under 300 LOC
  6. Test in same session ‚Äî "write tests for what you just built"
  7. Commit atomically ‚Äî small chunks, descriptive messages
  8. Verify on device ‚Äî simulator and/or physical device

AGENTS.md ‚Äî The Single Biggest Win steipete

Every project gets an AGENTS.md (or CLAUDE.md) at the root. This is the project's brain dump for the agent.

"I noticed Claude asks fewer clarifying questions and writes more accurate code."

What goes in AGENTS.md:

  • Project overview + architecture
  • File size rule (300 LOC max)
  • CLI commands: build, test, run, lint
  • Dependencies + how to install
  • Key patterns used in the codebase
  • Common pitfalls / "don't do this"
  • One-liner for each external tool: gh, psql, vercel, axiom

üèóÔ∏è Build Process

New Feature (Small, < 1 day)

Describe in plain English

Be specific about behavior, edge cases, UI expectations.

Plan first

"Plan this. Give me 3 options. Don't build yet." Pick the best, refine, THEN approve.

Build it

Type "y". Steer every few minutes. If a file passes 300 lines, tell it to split.

Write tests

"Write tests for what you just built. Check edge cases." Same session!

Commit

Run tests, verify in simulator, commit in logical chunks.

New Feature (Big, multi-day) steipete

Brain-dump into Gemini AI Studio

Free, 1M token context. Dictate via Wispr Flow. Find similar OSS and drag in via repo-to-text.

Generate a Software Design Document

Ask Gemini: "Write an SDD for this." Typically ~500 lines.

Two-Context Critique (the secret weapon)

Copy SDD ‚Üí fresh Gemini chat ‚Üí "20 things underspecified?" ‚Üí Copy answers back ‚Üí Repeat 3-5 rounds until questions are trivially niche.

Save spec, point agent at it

Save as docs/spec-[feature].md ‚Üí "Build spec.md"

Steer, test, ship

Steer the build. Write tests in same context. Commit surgically.

Post-Build Checklist Lex #491

After any major implementation, run this exact sequence:

  1. "Summarize intent in 3 bullets"
  2. "What would you refactor now?"
  3. "Which edge cases are under-tested?"
  4. "Write those tests"
  5. "Update docs"

Bug Fix

Paste exact error/crash log

Don't summarize ‚Äî paste verbatim. "Here's the error. Find and fix the root cause."

Write a regression test

"Write a test that would have caught this." Verify it fails without the fix.

Commit

git commit -m "fix: [concise description]"

Refactor / Cleanup

This is where parallel agents shine. Run 3-4 on different files ‚Äî low blast radius. steipete

Find files over 300 LOC

find . -name "*.swift" | xargs wc -l | sort -rn | head -20

Split with agents

"Split [file] into [logical pieces]." Use extension pattern: View+Section.swift

Run tests after each split, commit after each file

Atomic changes. One file at a time. Tests green before moving on.

Apple Docs Pipeline steipete

Before any Apple framework work, convert docs to markdown so agents can read them:

1. Go to https://llm.codes
2. Paste: https://developer.apple.com/documentation/[framework]
3. Download generated llms.txt
4. Save to docs/apple-docs/ in your project
5. Add to AGENTS.md so agent loads it into context

Pre-convert: SwiftUI, HealthKit (RunQuest), MapKit (RunQuest), SwiftData (GetSmart), StoreKit (both), CloudKit (GetSmart).

üîí Security & Reliability

Security Posture Lex #491

Pete confirmed: the biggest mistakes come from exposing gateway/backend publicly.

Non-negotiable rules:

  • Private network only ‚Äî no public exposure of debug/admin surfaces
  • mDNS off ‚Äî don't broadcast your presence
  • AllowFrom strict ‚Äî DMs locked to your user ID only
  • configWrites disabled ‚Äî no config changes via chat
  • High-capability models for privileged automation ‚Äî weak/cheap models are more prompt-injection susceptible Lex #491
  • File permissions hardened ‚Äî chmod 700 on .openclaw, chmod 600 on secrets

Safety net (Pete's approach): steipete

  • Hourly Arq snapshots
  • SuperDuper disk clone
  • Git commits as checkpoints
  • After two months of dangerous mode: zero incidents

Gateway & Heartbeat Our experience

Critical architecture rules (burned from real incidents):

  • NEVER manage gateway manually ‚Äî launchd owns it (KeepAlive: true)
  • NEVER write raw config ‚Äî always gateway config.patch
  • NEVER put agent settings at root level ‚Äî they go under agents.defaults
  • NEVER run custom background daemons ‚Äî use cron/heartbeat
  • NEVER retry failed config in a loop ‚Äî read error, fix, try ONCE

Heartbeat discipline:

  • Every 30min during active hours (7AM-11PM)
  • Lean checks only: gateway health + context pressure
  • No auto-starting background processes from heartbeat
  • If nothing needs attention: HEARTBEAT_OK

Cron over custom daemons: Our experience

We learned the hard way: a custom monitor-runner.sh caused 18 gateway restarts and rate limits on both providers. Replaced with OpenClaw's built-in cron system. Each concern owned by exactly one system (MECE).

üì± App Playbooks

The 300 LOC Discipline steipete

Hard limit: no production Swift file should exceed 300 LOC. Agents "freak out" and lose reliability as files grow.

Split patterns (proven on RunQuest):

Pattern 1: Swift Extensions

// ProfileView.swift (main, < 300 LOC)
struct ProfileView: View {
    @State internal var selectedTab: ProfileTab = .stats
    var body: some View { /* main layout */ }
}

// ProfileView+Stats.swift
extension ProfileView {
    var statsSection: some View { ... }
}

// ProfileView+Account.swift
extension ProfileView {
    var accountSection: some View { ... }
}

Key: Change private ‚Üí internal on @State/@Environment for cross-file access.

Pattern 2: Extract ViewModel

// Before: FatView.swift (800 LOC)
// After:
//   FatView.swift (200 LOC, UI only)
//   FatViewModel.swift (200 LOC, state + logic)
//   FatView+Sections.swift (200 LOC, subviews)

Audit commands:

# 300+ offenders
rg --files -g '*.swift' . \
| while IFS= read -r f; do
    n=$(wc -l < "$f" | tr -d ' ')
    [ "$n" -gt 300 ] && printf "%6s  %s\n" "$n" "$f"
done | sort -nr

Multiple Terminals at Once

"When I'm not refactoring I usually run 1-2 agents; for cleanup/tests/UI work ~4 seems to be the sweet spot."
  • While one agent thinks, switch to another and give it the next task
  • Parallel streams: tests in one, feature in another, docs in a third
  • Pick non-overlapping areas of the codebase

Spec-First Development

  1. Research ‚Äî Paste ideas into Gemini AI Studio. Find similar OSS via repo-to-text.
  2. Generate SDD ‚Äî Gemini converts ideas into a ~500 line Software Design Document.
  3. Two-Context Critique ‚Äî Copy SDD ‚Üí fresh chat ‚Üí "20 things underspecified?" ‚Üí Copy back ‚Üí Repeat 3-5 rounds.
  4. Build ‚Äî Save as docs/spec.md, open Claude Code: "Build spec.md"

Testing Philosophy

  • Tests AFTER the feature ‚Äî "Agents are great at back-filling tests once the shape exists"
  • Same context ‚Äî "Adding tests per feature is much better than trying to add them at the end"
  • Tests are MORE important with agents ‚Äî "You can't trust them. They'll eventually break stuff."

Proof: VibeMeter ‚Äî 28,599 LOC, 86% test coverage, 628 test methods. Built in 3 days.

MCP vs CLI

"Even removed my last MCP, since Claude sometimes would go off spinning up Playwright unasked when it could simply read the code."

Pete built 5+ MCPs ‚Üí removed ALL of them. CLIs produce less noise, less context pollution.

Lex #491 Key Insights Lex #491

  • The Agentic Trap ‚Äî Over-orchestrating with 18+ slash commands. Return to short, high-intent prompts once the agent understands.
  • Intent check first ‚Äî "Do you understand the intent of this change?" before building.
  • Post-build refactor loop ‚Äî "What would you refactor now?" + edge-case tests + docs.
  • Model evaluation horizon ‚Äî Give a new model ~1 week before judging.
  • Security posture matters ‚Äî Private network, avoid weak models for privileged tasks.
  • Apps become APIs ‚Äî Agent-native world favors API-first, automation-friendly products.
  • "Builder" > "Programmer" ‚Äî Role transformation, not elimination.

üí∞ Product & GTM

Apps Become APIs Lex #491

Pete's strongest business thesis: many app categories lose direct UX primacy. Agent-native world favors API-first and automation-friendly products.

  • Keep features composable ‚Äî services, not just UI flows
  • Preserve clean service boundaries and automation hooks
  • Think about how an agent would use your app's functionality
  • Internal APIs are a feature, even if users never see them

Monetization Strategy Our experience

  • RevenueCat for subscription management (both apps)
  • StoreKit 2 for native paywalls
  • PostHog for analytics and feature flags
  • Sentry for crash reporting
  • Agent-built features ship faster ‚Üí faster iteration on pricing/packaging
  • Use agents to analyze user feedback emails and extract feature requests
